# 数据层架构设计

## 一、层定位

数据层（Data Layer）是整个系统的"记忆与知识底座"，负责学生画像、知识库、资源库三大类核心数据的存储、管理、检索与治理。该层向上为智能体层与模型层提供高质量、结构化的数据支撑；向下对接具体的数据库、对象存储、搜索引擎等基础设施，确保数据的一致性、安全性与可扩展性。

---

## 二、设计目标

1. **数据一体化**：打通学生行为数据、知识内容数据、教育资源数据，消除数据孤岛。
2. **实时与离线兼顾**：支持毫秒级的画像查询与秒级的复杂分析任务。
3. **语义检索能力**：支持基于自然语言的知识库与资源库语义搜索，为 RAG 与智能推荐提供基础。
4. **隐私与合规**：严格遵守教育数据隐私法规，实现数据分级、脱敏、审计与可控访问。

## 四、核心数据域

### 4.1 学生画像数据域（Student Profile Domain）

#### 4.1.1 数据分类

| 数据类别 | 具体内容 | 存储方案 | 更新频率 |
|---------|---------|---------|---------|
| **基础属性** | 用户 ID、学段、年级、目标考试、注册时间 | MySQL | 低频 |
| **六大画像维度** | 学业阶段、信息偏好、学习节奏、认知风格、社交需求、技术接受度 | MySQL + Redis | 中频 |
| **学习行为日志** | 页面浏览、视频观看时长、题目作答记录、搜索关键词、Agent 对话历史 | ClickHouse + Kafka | 实时 |
| **知识掌握状态** | 各知识点的掌握度评分、熟练度曲线、遗忘曲线参数 | MySQL + Redis | 高频 |
| **错题档案** | 错题原题、错误答案、错误类型标签、订正记录、关联知识点 | MySQL + 对象存储 | 中频 |
| **社交关系** | 学习小组、关注关系、互动记录 | MySQL | 中频 |

#### 4.1.2 画像计算与更新机制
- **实时更新**：答题正确/错误、学习时长累计等行为即时更新 Redis 中的热数据。
- **定时更新**：每晚离线任务基于全天行为日志，重新计算画像维度得分与知识掌握度，写入 MySQL。
- **事件驱动**：关键行为（如连续 3 次同一知识点错误）触发画像标签变更与智能体主动干预。

#### 4.1.3 画像输出接口
```typescript
interface StudentProfile {
  userId: string;
  basic: { grade: string; targetExam?: string; };
  dimensions: {
    academicStage: 'exam-sprint' | 'skill-up' | 'interest-explore' | 'academic-consolidate';
    infoPreference: 'self-search' | 'kol-follow' | 'community' | 'platform-immersive';
    learningPace: 'planned' | 'fragmented' | 'deadline-driven' | 'spontaneous';
    cognitiveStyle: 'visual' | 'auditory' | 'kinesthetic' | 'reading';
    socialNeed: 'competitive' | 'collaborative' | 'independent' | 'mentor-dependent';
    techAcceptance: 'tool-geek' | 'minimalist' | 'traditional' | 'ai-early-adopter';
  };
  knowledgeState: Record<string, { mastery: number; lastReview: Date; stability: number }>;
  weakPoints: string[];      // 薄弱知识点 ID 列表
  interests: string[];       // 兴趣标签
  updatedAt: Date;
}
```

### 4.2 知识库数据域（Knowledge Base Domain）

#### 4.2.1 数据分类

| 数据类别 | 具体内容 | 存储方案 | 说明 |
|---------|---------|---------|------|
| **学科知识体系** | 课程标准、章节结构、知识点层级关系、前置后置依赖 | MySQL + Neo4j（图数据库） | 构成知识图谱骨架 |
| **知识点内容** | 概念定义、公式定理、典型例题、常见误区、记忆技巧 | MySQL + 对象存储 | 富文本/视频/图片 |
| **题库** | 题目内容、答案、解析、难度评级、知识点标签、能力维度标签 | MySQL + Elasticsearch | 支持多维检索 |
| **向量索引** | 所有文本内容的 Embedding 向量 | Milvus / Pinecone | 语义检索与 RAG |

#### 4.2.2 知识图谱构建
- **节点类型**：学科 → 模块 → 章节 → 知识点 → 微技能
- **边类型**：`包含`（层级）、`前置需要`（依赖）、`关联`（扩展）、`易混淆`（对比）
- **应用场景**：
  - 智能体定位学生知识盲区时，沿`前置需要`边追溯根源；
  - 推荐 Agent 沿`关联`边推送拓展内容；
  - 规划 Agent 基于拓扑排序生成符合认知顺序的学习路径。

#### 4.2.3 RAG（检索增强生成）支持
- **检索流程**：
  1. 用户 query → Embedding 模型向量化；
  2. 向量数据库语义召回 Top-K 相关知识点/题目；
  3. Elasticsearch 关键词补充召回；
  4. 重排序（Rerank）模型精排；
  5. 将检索结果作为上下文注入大模型生成回答。
- **更新机制**：新增或修改知识点后，自动重新计算 Embedding 并更新向量库。

### 4.3 资源库数据域（Resource Library Domain）

#### 4.3.1 数据分类

| 数据类别 | 具体内容 | 存储方案 |
|---------|---------|---------|
| **课程资源** | 视频课程、直播回放、课件 PPT、讲义文档 | 对象存储 + MySQL 元数据 |
| **文档资料** | 笔记、思维导图、真题试卷、模拟题、电子书 | 对象存储 + Elasticsearch |
| **外部链接** | 经过审核的优质第三方资源 URL、博主推荐 | MySQL |
| **资源评价** | 用户评分、评论、收藏数、完课率、学习转化率 | MySQL + ClickHouse |

#### 4.3.2 资源元数据标准
```typescript
interface Resource {
  resourceId: string;
  title: string;
  type: 'video' | 'document' | 'quiz' | 'live' | 'link';
  subject: string;
  topics: string[];          // 关联知识点 ID
  difficulty: 1 | 2 | 3 | 4 | 5;
  duration: number;          // 预计学习时长（分钟）
  format: string[];          // 内容形式（视频/图文/音频）
  targetDimensions: string[]; // 适合的学生画像维度标签
  rating: number;
  usageCount: number;
  source: string;            // 内部制作 / 外部引入 / UGC
  vectorId?: string;         // 关联的向量索引 ID
  createdAt: Date;
}
```

#### 4.3.3 资源推荐数据支持
- **协同过滤数据**：用户-资源交互矩阵（浏览、收藏、完成、评分），存储于 ClickHouse，每日更新推荐模型。
- **内容相似度**：基于资源标题、简介、知识点的 Embedding 向量，计算内容间相似度。
- **画像匹配度**：资源的目标画像维度与学生当前画像的匹配分数。

## 五、数据治理中心

### 5.1 数据质量管理
- **完整性校验**：每日扫描缺失必填字段的记录，自动修复或告警。
- **一致性校验**：跨库数据同步状态监控（如 MySQL 与向量库的知识点版本一致性）。
- **准确性校验**：抽样人工审核模型生成的知识点内容，建立质量评分体系。

### 5.2 数据安全与隐私
- **分级分类**：
  - L1 公开：学科知识体系、公共资源元数据；
  - L2 内部：聚合后的学情统计、匿名行为数据；
  - L3 敏感：学生个人画像、原始行为日志、错题内容；
  - L4 机密：身份信息、支付信息。
- **脱敏策略**：L3/L4 数据对外提供时必须脱敏或走差分隐私处理。
- **访问控制**：RBAC（基于角色的访问控制）+ ABAC（基于属性的访问控制），智能体层调用数据需通过权限网关。
- **审计日志**：所有数据访问与修改操作记录不可篡改的审计日志，保留 180 天。

### 5.3 数据生命周期
- **热数据**：最近 7 天的行为日志、活跃会话，存储于 Redis / ClickHouse。
- **温数据**：近 1 年的历史数据，存储于 MySQL / 对象存储标准存储。
- **冷数据**：超过 1 年的归档数据，迁移至低成本对象存储（如阿里云 OSS 归档型）。

## 六、语义检索引擎

### 6.1 混合检索架构
```
用户 Query
    │
    ▼
┌─────────────┐
│  Query理解   │ → 意图分类、实体识别、同义词扩展
└─────────────┘
    │
    ├──────────────┬──────────────┐
    ▼              ▼              ▼
 向量检索      关键词检索      图谱检索
(Milvus)   (Elasticsearch)   (Neo4j)
    │              │              │
    └──────────────┴──────────────┘
                   │
                   ▼
            ┌─────────────┐
            │  重排序模型   │ → 精排后返回 Top-N
            └─────────────┘
```

### 6.2 检索场景映射

| 场景 | 主检索方式 | 辅助检索方式 |
|-----|-----------|------------|
| 学生搜题 / 答疑 | 向量检索（题目语义） | 关键词检索（公式符号） |
| 资源推荐 | 画像匹配 + 协同过滤 | 内容向量相似度 |
| 知识盲区定位 | 图谱检索（依赖路径） | 学生画像状态过滤 |
| 错题同类题推荐 | 向量检索（题目语义相似） | 知识点标签精确匹配 |

## 七、实时计算引擎

### 7.1 技术选型
- **流处理**：Apache Flink / Kafka Streams
- **应用场景**：
  - 实时学习时长统计与在线状态更新；
  - 实时答题正确率监控与即时薄弱点标记；
  - 实时排行榜与成就触发。

### 7.2 典型数据流
```
学生答题事件 → Kafka → Flink 窗口计算 → 更新 Redis 实时状态 → 触发智能体干预
                │
                └──→ 写入 ClickHouse 明细 → 夜间离线聚合 → 更新 MySQL 画像
```

## 八、与上层交互

```
数据层  <--SQL / ORM-->  智能体层（画像查询、知识库检索、资源推荐）
数据层  <--Vector Search-->  模型层（RAG 上下文检索、语义匹配）
数据层  <--REST / GraphQL-->  表现层（个人档案、学习报告、排行榜）
```
