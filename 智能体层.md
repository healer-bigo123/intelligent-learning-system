# 智能体层架构设计

## 一、层定位

智能体层（Agent Layer）是整个系统的"大脑中枢"，负责多智能体的协同调度、任务分解、状态管理与结果聚合。该层向上为表现层提供统一、连贯的智能服务接口；向下调用模型层的大模型能力与数据层的学生画像、知识库、资源库，实现"感知-决策-执行-反馈"的闭环。

---

## 二、设计目标

1. **多智能体协同**：不同职能的智能体可独立运行，也可根据复杂任务自动组队协作。
2. **动态任务编排**：支持基于学生画像与实时场景的动态工作流生成，而非硬编码流程。
3. **状态持久化**：长时任务（如一周学习计划制定）可在任意节点中断并恢复。
4. **可观测与可干预**：智能体的思考过程、工具调用记录对用户透明，支持人工纠正与接管。

## 四、调度中心核心组件

### 4.1 意图识别器（Intent Recognizer）
- **职责**：解析用户自然语言输入，识别真实意图与隐含需求。
- **输入**：用户 query + 当前页面上下文 + 学生画像摘要。
- **输出**：结构化意图对象 `{ intentType, entities, urgency, preferredAgent, emotionTag }`。
- **示例**：
  - 输入："最近数学导数总是错，怎么办"
  - 输出：`{ intentType: "求助", entities: { subject: "数学", topic: "导数", issue: "易错" }, urgency: "高", preferredAgent: "答疑Agent", emotionTag: "焦虑" }`

### 4.2 任务分解器（Task Decomposer）
- **职责**：将复杂意图拆解为可并行或串行的子任务，分配给合适的智能体。
- **策略**：
  - 简单问答：单智能体直接响应；
  - 复杂规划：规划Agent 制定框架 → 推荐Agent 填充资源 → 答疑Agent 预生成 FAQ；
  - 紧急求助：高优先级插队，必要时唤醒人工老师 Agent。
- **输出**：有向无环图（DAG）形式的任务执行计划。

### 4.3 状态管理器（State Manager）
- **职责**：维护每个会话、每个长时任务的状态机。
- **状态定义**：`idle → analyzing → dispatching → executing → reviewing → completed / failed`
- **持久化**：使用 Redis 存储活跃状态，MySQL 归档历史会话，支持断点续传。

### 4.4 记忆检索器（Memory Retriever）
- **职责**：在任务执行前，为智能体检索相关上下文记忆。
- **记忆来源**：
  - **短期记忆**：当前会话历史（最近 10 轮对话）。
  - **长期记忆**：学生画像中的学习偏好、历史错题、知识薄弱点。
  - **外部记忆**：知识库中的相关概念、资源库中的关联资料。
- **检索技术**：向量数据库（Milvus/Pinecone）语义检索 + 关键词召回混合策略。

### 4.5 结果聚合器（Result Aggregator）
- **职责**：收集多智能体的子任务结果，去重、排序、格式化后返回表现层。
- **冲突处理**：当多个 Agent 给出矛盾建议时，触发冲突仲裁器，基于置信度与学生画像做最终决策。

### 4.6 冲突仲裁器（Conflict Arbiter）
- **职责**：解决多智能体间的意见冲突或资源竞争。
- **策略**：规则引擎（硬性约束优先）+ 大模型推理（软性建议综合）。

## 五、智能体集群定义

### 5.1 答疑 Agent（Q&A Agent）
- **职能**：回答学科问题、解释概念、梳理解题思路。
- **工具集**：知识库检索、题库检索、公式渲染、逐步推理（CoT）。
- **特殊能力**：识别学生提问中的知识盲区，主动关联前置知识点。

### 5.2 规划 Agent（Planning Agent）
- **职能**：制定日/周/月学习计划，动态调整优先级。
- **工具集**：学生画像读取、资源库检索、时间块分析、目标拆解（OKR 方法）。
- **特殊能力**：结合学生的时间管理画像（规律计划型 / 碎片突击型等），生成可执行的计划。

### 5.3 批改 Agent（Grading Agent）
- **职能**：批改主观题、作文、代码，给出评分与改进建议。
- **工具集**：多模态模型（图像识别手写内容）、代码执行沙箱、评分 rubric 库。
- **特殊能力**：不仅指出错误，还能追溯错误类型（概念不清 / 粗心 / 表达不规范）。

### 5.4 陪伴 Agent（Companion Agent）
- **职能**：情感支持、学习动力激励、焦虑缓解。
- **工具集**：情绪识别模型、正向反馈语料库、成长可视化数据。
- **特殊能力**：识别学生情绪状态（通过文字语气、学习行为异常检测），主动发起关怀对话。

### 5.5 推荐 Agent（Recommendation Agent）
- **职能**：推荐学习资源、同类题、拓展阅读、学习搭子。
- **工具集**：协同过滤算法、内容向量匹配、学生画像相似度计算。
- **特殊能力**：解释推荐原因（"因为你上周在函数单调性上错了 3 次，推荐这道变式题"）。

### 5.6 分析 Agent（Analytics Agent）
- **职能**：学情分析、知识图谱构建、学习效果归因。
- **工具集**：数据层聚合查询、知识图谱推理、统计模型。
- **特殊能力**：自动生成阶段性学习报告，指出进步与退步维度。

## 六、智能体间协作模式

### 模式 1：串行流水线（Sequential Pipeline）
适用场景：复杂任务需分阶段处理。
示例：学生上传一张错题照片 → 批改Agent 识别题目与错误 → 答疑Agent 讲解正确思路 → 推荐Agent 推送 3 道同类题 → 规划Agent 将同类题加入明日复习清单。

### 模式 2：并行投票（Parallel Voting）
适用场景：需要多角度验证的问题。
示例：学生问"这道题的解法是否最优" → 同时分发给 2 个答疑Agent（分别采用不同推理策略）→ 结果聚合器对比一致性 → 返回最优解并标注争议点。

### 模式 3：主从协作（Master-Slave）
适用场景：一个 Agent 主导，其他 Agent 提供信息支持。
示例：规划Agent 制定周计划时，主动向分析Agent 查询薄弱知识点、向推荐Agent 获取推荐资源列表，再综合生成计划。

### 模式 4：人机协同（Human-in-the-Loop）
适用场景：高 stakes 决策或 Agent 置信度不足时。
示例：规划Agent 生成的冲刺计划涉及每天学习 12 小时，触发人工审核节点，由真人老师或学生本人确认后生效。

## 七、技术实现要点

| 组件 | 技术选型 | 说明 |
|-----|---------|------|
| 智能体框架 | LangChain / LlamaIndex + 自研调度层 | 利用框架的 Tool/Chain 能力，自研部分解决多 Agent 协作 |
| 工作流引擎 | Temporal / Cadence | 支持长时、可靠、可观测的任务编排 |
| 消息队列 | RabbitMQ / Apache Kafka | 智能体间异步通信与事件驱动 |
| 状态存储 | Redis（热数据）+ MySQL（冷数据） | 会话状态与历史归档 |
| 向量检索 | Milvus / Pinecone | 记忆检索与知识库语义匹配 |
| 可观测性 | OpenTelemetry + Jaeger + Prometheus | 追踪每个请求在多 Agent 间的流转路径 |

## 八、与下层交互

```
智能体层  <--Function Call / API-->  模型层（请求大模型推理、多模态生成）
智能体层  <--SQL / ORM / Vector Search-->  数据层（读写学生画像、知识库、资源库）
智能体层  <--WebSocket / REST-->  表现层（推送结果、接收用户反馈）
```
